Object Oriented Programming (shortened as OOP) is a programming concept based on real world objects. The belief is that we can effectively create anything we see in the real world in code and better. Some of its main concepts (which were focused on in this course) are Encapsulation, Inheritance, Polymorphism, and Abstraction. These ideas are a foundation, and object oriented programming languages such as Java☕, Python🐍, C++ ©️ or C# #️⃣support this with memory allocation, references and effectively a complex system of transisters that take in input, and produce output.

Why might one even go through the hassle of creating so called "△ objects ◎"? Benefits include code-reusability, code-maintainability, a well structured system, security, flexibility, and more.

First, lets take a very brief look at what the four main OOP topics taught in this course means conceptually.

- Encapsulation

    Encapsulation refers to having variables (attributes) and methods (functions/abilities) bundled up into a singular unit (class) which may have the ability to be private and/or gatekeep certain information. It helps achieve data-hiding. Its internal details are hidden from outside itself, but it allows for interactions which are black-boxed and kept safe.

- Inheritance

    Inheritance is the ability for a child object to "inherit" or get passed on traits, attributes, and abilities from a "parent" object. This enables the new derived class to have the relationship "is-a" with it's base class + have its own unique factors - kind of like an evolution.

- Polymorphism

    Polymorpism is the ability for something to have different forms. In computer science, it is specifically children classes implementing and morphing its own unique features with the parent class. In Java, it is said that there is an Object class at the top of the class hierachy. From that, there are childrens all of which have unique features some concrete, some abstract, together creates the programs that we use every day.

- Abstraction

    Abstraction is the idea that you can have literally anything as an object. You can even have abstract and unreal ideas, those of which that may be abstract classes or interfaces which as ideas such as so called variables and methods, but are not properly/literally defined. This gives programmers the ability to start with any concept they can think of and work there way "down" by slowly adding definitions to their idea. This promotes code code-maintainability, organization and helps code become scalable.

    Encapsulation Example

As stated previously, Encapsulation is essentially the ability for a chunk of code (in this case a class) to be self contained and have data-hiding. The class allows for some information to be directly given while others to be protected. Lets adjust our Bird class to implement this idea!

Now all variables are not directly accessible from external sources. Private class variables may be accessed from everywhere within its own class but not from others.
Object.VariableName; ←← now leads to a syntax error.

    Polymorphism Example

As we know from inheritence, subclasses inherit the functions and attributes of their superclass. The idea of Polymorphism and its implementations in Java build on this and allow subclasses to redefine what superclass' method does. This essentially allows a singular object to have multiple forms depending on the level of object it is.

Say we have a superclass Mammal, and subclass Bird. (Refer to the organism tree I made at the bottom of Inheritance). All (techincally most) mammals can travel in some shape or form. For simplicity purposes say a mammal can travel by foot. The subclass Bird may redefine travelling with its special wings, but it can also travel by foot.

When a Bird object is asked to travel, it will default to travelling by flight, but if the Bird class allows itself to travel using it's superclass, it can via the keyword super.

Lets see this in code. Here is a Mammal class:


        File Name: Mammal.java

    1     public class Mammal extends Animal {
    2       // Attributes / Variables
    3       // Functions / Methods
    4       public void travel(){
    5           System.out.println("Travelling by foot");
    6       }
    7     }

Here is the Bird subclass showcasing how it may redefine the travel method.


        File Name: Bird.java

    1     public class Bird extends Mammal {
    2       // Attributes / Variables
    2       // Functions / Methods
    3       public void travel(){
    4           System.out.println("Travelling with wings");
    5       }
    6     }

Now when a bird object performs the travel functions, this happens:


        (IN ANOTHER FILE)

    1     Bird var = new Bird();
    2     var.travel()

        (Output Console)
    1     Travelling with wings

The technique of a subclass redefining a superclass is called OVERRIDING.

Here is a Bird class that instead calls its superclass's method, then adds its unique function. It calls the superclass method via the keyword "super"


        File Name: Bird.java

    1     public class Bird extends Mammal {
    2       // Attributes / Variables
    2       // Functions / Methods
    3       public void travel(){
    4           super.travel();
    5           System.out.println("Travelling with wings");
    6       }
    7     }

Here is a demonstration:


        (IN ANOTHER FILE)

    1     Bird var = new Bird();
    2     var.travel()

        (Output Console)
    1     Travelling by foot
    2     Travelling with wings

The "super" keyword essentially calls the superclass of the class it is in.

Some additional things you need to know are:

  • If you have
    SuperClass var = new SubClass();
    var may NOT use SubClass methods unless it is downcasted or is a overridden method. The overridden method only works because the method name exists within SuperClass allowing to be passed during Compile Time, then at runtime recognizes the SubClass overrider.
  • A SubClass may not create a SuperClass.
  • If a SuperClass calls a method within itself, the SuperClass's method is used, not the SubClass's, even if the name overridden.

    Abstraction Example

As stated previously, Abstraction is the idea that classes may be abstract. Classes may have predetermined methods, but no definition/code within the method - making it essentially useless. To add use to such a vaguely defined class, it has to be extended with subclasses which purposefully overrides the methods and gives it functions.

Here is a class called Solid:

   10       }

There are many different vriable names and method names, but not definition. The "abstract" keyword ensures that this class may not be instantiated and allows for different syntax.
